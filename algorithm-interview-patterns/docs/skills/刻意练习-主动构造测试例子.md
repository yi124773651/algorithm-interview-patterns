# 刻意练习：主动构造测试例子

## 概览
这份技能不是教你“多写几个例子”，而是教你建立一套稳定的心理表征：
**把测试例子当成理解问题结构、暴露思维漏洞、验证算法边界的工具。**

核心原则只有一句：
**不是想到什么测什么，而是先找风险点，再按风险造例。**

## 什么时候使用
- 做算法题时，发现自己只会跑题目给的样例
- 写完代码后觉得“应该没问题”，但说不出最容易错在哪
- 遇到 bug 时，不知道该构造什么输入去复现
- 讲题时能讲思路，但讲不出为什么某些边界一定要测
- 复盘错题时，只记住了正确代码，没有沉淀“当时应该怎么造例”

## 你要建立的心理表征

### 1. 输入空间表征
不要只看“这是一道什么题”，而要看“输入有哪些可以变化的维度”。

常见维度：
- 大小：最小值、最大值、极端长度
- 结构：空、单个、链式、嵌套、分段、重复
- 位置：前缀、中间、后缀、首尾相邻
- 数值：负数、零、正数、重复值、多位数、溢出边缘
- 分布：全相同、严格递增、严格递减、随机、稀疏、密集
- 关系：相等、交叉、包含、重叠、断开

你的目标不是记住所有维度，而是形成条件反射：
**看到题，先问“这题的输入有哪些维度会变化？”**

### 2. 风险点表征
不要只想“这段代码在做什么”，要想“它最容易在哪些地方错”。

常见风险点：
- 边界没处理：最小、最大、空、单元素
- 状态没重置：跨轮次污染、跨层级污染
- 位置丢失：前缀、后缀、中间拼接出错
- 多位解析错误：只处理一位数
- 嵌套或递归层级错误
- 重复元素、相等元素处理错误
- 单调结构在全升/全降时出错
- 只验证主路径，没有验证反路径

你的目标是形成第二个条件反射：
**每写一个关键分支，就问“这一支最怕什么输入？”**

### 3. 算法不变量表征
强者构造测试例子，常常不是按“感觉”，而是按不变量。

你要学会问：
- 这个算法每一步必须保证什么？
- 哪些输入会让这个保证最容易被破坏？
- 如果不变量被破坏，输出会呈现什么错误症状？

例子：
- 栈题：栈里维护的到底是什么，什么时候应该弹出
- 滑动窗口：窗口内统计量什么时候增加、什么时候减少
- 二分：左闭右闭还是左闭右开，边界推进是否死循环
- 动态规划：状态定义是否覆盖 base case，转移是否重复或遗漏

### 4. 反例驱动表征
真正高效的测试，不是“证明自己对”，而是“主动攻击自己的当前思路”。

你要建立这种习惯：
- 我这份代码最脆弱的 3 个点是什么？
- 每个脆弱点，能不能造一个一击致命的输入？

这叫反例驱动，而不是样例驱动。

## 通用造例框架
每道题，至少按下面 5 类构造：

### 1. 正常样例
作用：验证主流程能跑通。

问题模板：
- 题目给的标准样例过了吗？
- 一个最典型、最符合预期的输入是什么？

### 2. 最小样例
作用：攻击初始化、边界判断、空分支。

问题模板：
- 最小合法输入是什么？
- 单元素时逻辑还成立吗？

### 3. 边界样例
作用：攻击极值、长度极限、特殊位置。

问题模板：
- 最大值、最小值、全相同、全递增、全递减呢？
- 首元素、尾元素参与关键逻辑时会不会出错？

### 4. 打脸样例
作用：专门攻击你当前写法的薄弱点。

问题模板：
- 我现在这版代码最可能在哪一行挂掉？
- 哪个输入能直接让这个漏洞暴露出来？

### 5. 组合样例
作用：验证多个特征叠加时，状态管理是否稳定。

问题模板：
- 如果“嵌套 + 多位数 + 前后缀”同时出现会怎样？
- 如果多个边界同时叠加，代码还能稳吗？

## 实战流程
每次做题，按这 6 步走。

### 第一步：先写“输入维度”
不要先写代码，先写 3 到 5 个会变化的维度。

模板：
- 这题的关键输入维度有：______、______、______。

### 第二步：再写“风险点”
先预测自己会错在哪。

模板：
- 这题我最容易错的 3 个点：
- 1. ______
- 2. ______
- 3. ______

### 第三步：每个风险点对应一个测试
不要泛泛造例，要一一映射。

模板：
- 风险点 A -> 测试例子 A
- 风险点 B -> 测试例子 B
- 风险点 C -> 测试例子 C

### 第四步：代码里的每个关键分支都配一个念头
看到代码结构，就自动触发测试问题。

模板：
- 看到解析数字 -> 我有没有测多位数？
- 看到入栈/入队 -> 我有没有测层级或顺序变化？
- 看到弹栈/结算 -> 我有没有测边界触发？
- 看到循环结束后处理 -> 我有没有测“最后一段”是否被漏算？

### 第五步：用“反例优先”排序
先跑最容易打脸当前思路的例子，再跑正常样例。

原因：
- 正常样例容易让你产生虚假安全感
- 反例更能暴露理解缺口

### 第六步：做完题后补一句复盘
每题只复盘一句最关键的话。

模板：
- 这题以后最该主动构造的例子是：______。
- 因为它专门攻击：______。

## 按题型建立半自动反射

### 栈 / 单调栈
优先问：
- 全递增会不会漏结算？
- 全递减会不会过度弹栈？
- 相等元素怎么处理？
- 最后一轮是否需要额外清算？

### 双指针 / 滑动窗口
优先问：
- 左指针不动、右指针猛进时会怎样？
- 左右同时移动时会不会漏答案？
- 窗口缩小时统计量是否正确回退？
- 满足条件和不满足条件的临界点是否测过？

### 二分搜索
优先问：
- 最左、最右、恰好不存在时怎么返回？
- 中点取整是否死循环？
- 左闭右闭和左闭右开是否一致？

### 递归 / 回溯
优先问：
- 终止条件是否会少一层或多一层？
- 路径状态是否污染下一条分支？
- 去重在树层还是树枝？

### 动态规划
优先问：
- base case 是否完整？
- 状态转移是否遗漏一种来源？
- 一维压缩后，更新顺序会不会污染旧状态？

## 用 LeetCode_394 演示这套方法
题目：字符串解码。

### 输入维度
- 是否嵌套
- 重复次数是否多位数
- 是否有前缀或后缀普通字符
- 是否有多段拼接

### 风险点
- 只处理一位数
- 只处理括号内字符串，漏掉前后缀
- 层级状态污染

### 风险对应的例子
- 多位数风险 -> `10[a]`
- 前后缀风险 -> `abc3[cd]xyz`
- 嵌套风险 -> `3[a2[c]]`
- 组合风险 -> `2[ab3[c]]x`

这就是“按风险造例”，而不是“随便想几个例子”。

## 每日刻意练习模板
建议连续练 14 天，每天 3 题，不要求都写代码，但每题都必须写测试思路。

### 训练动作
- 第 1 分钟：写输入维度
- 第 2 分钟：写 3 个风险点
- 第 3 分钟：给每个风险点配一个例子
- 第 4 分钟：排序，先跑最打脸的
- 第 5 分钟：写一句复盘

### 训练目标
- 第一阶段：能稳定写出 5 类测试
- 第二阶段：能让每个测试都指向一个风险点
- 第三阶段：看到代码分支时，脑中自动浮现对应例子

## 自检清单
做完一道题，问自己：
- 我有没有先写输入维度，而不是直接写代码？
- 我有没有写出最小样例和边界样例？
- 我有没有专门攻击我当前思路的打脸样例？
- 我的每个关键测试，是否都能对应一个明确风险点？
- 我有没有复盘“下次遇到同类题，先测什么”？

## 一句口令
每次开始前，先对自己说：

**先找维度，再找风险；先造反例，再跑样例。**
